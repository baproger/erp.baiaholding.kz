# BAIA ERP / CRM — Документация проекта

> Единый файл-справочник: что построено, как работает, доступы, бизнес-правила.

## 1. Технологии
- **Backend:** Laravel 13 (PHP 8.5), MySQL `baia_erp`
- **Frontend:** Inertia.js + Vue 3 + Tailwind + Vite (Laravel Breeze)
- **Права:** spatie/laravel-permission (RBAC). **Admin — глобальный суперпользователь** (worksInCompany=true во всех фирмах). Роль admin назначает только admin; последний активный админ защищён от разжалования/деактивации.
- **Таймзона:** Asia/Almaty (+05:00); **Локализация:** ru / kk
- **Тесты:** PHPUnit — 162, все зелёные
- **Продакшн (Plesk):** сессии/кэш — файлы, очередь — sync (форсируется в config при APP_ENV=production, снижает нагрузку на БД); `vendor/` и `public/build/` закоммичены (на сервере нет composer/node); деплой — `DEPLOY.md`.

## 2. Роли и доступы
| Роль | Что видит |
|------|-----------|
| **admin** | всё |
| **director** | всё (наблюдатель), создаёт группы в чате |
| **financist** | финансы, аналитика, зарплата (сделки/цех — просмотр) |
| **manager** | ТОЛЬКО свои сделки/цех/финансы/бонус/дашборд; общей аналитики НЕ видит |
| **employee (цех)** | ВСЕ карточки цеха (наблюдатель, двигает этапы), свои задачи; сделки/суммы НЕ видит |

> ⚠️ **Репозиторий публичный — в документации и коде НЕТ логинов/паролей, ключей и реальных данных.** Сидер (`DatabaseSeeder`) создаёт первую учётку администратора с временным паролем — **сменить его при первом входе**. Реальные доступы, `.env`, пароль БД, `APP_KEY` хранятся ТОЛЬКО на сервере и в GitHub не попадают (см. `.gitignore`).

## 3. Модули
- **Сотрудники / Отделы:** CRUD, роли, деактивация.
- **Контрагенты:** в коде, скрыты из меню.
- **Сделки:** канбан+список; единый фильтр-бар (поиск компания/№/лот/договор, менеджер — только руководству, этап, срок с—по, счётчик найденных); карточка (вкладки: Информация, Задачи, Финансы, Документы, Комментарии, Доп.поля, История, Чат сделки). Поля: Компания (BAIA/ASU)*, Название компании (клиента)*, Адрес*, Номер договора (колонка bin), Дата договора, Источник-портал (ОМ/ЗЦП/ИОИ/СК/СК-ЭМ/СК-store/ОТП), Наименование товара (колонка client_name)*, Количество (колонка lot_number) + ед.изм. (штук/рулон/Комплект/Работа/метр/метр погонный), Сумма*, Срок, Заметка. Кнопки: Далее, Отправить в цех, Изменить, смена ответственного. **Поле «Название сделки» убрано:** везде (шапка карточки, списки сделок/цеха, ABC-аналитика, имя заказа цеха) показывается название компании; колонка deals.name осталась и зеркалит company_name при create/update (легаси-сделки: фронт показывает company_name || name).
- **Цех (Projects):** общий канбан; этапы Кесу…Оплата (редактируемые); финансы/история из исходной сделки; суммы скрыты от цеха.
- **Склад:** у каждой компании свой (materials + material_receipts). Приход оформляет admin/financist (director — просмотр); менеджер видит остатки; цех — нет доступа. Страница /warehouse: таблица «как в тетради» — материал, ед.изм, цена за ед., поступление (+qty, зелёным), сумма поступления (по ценам приходов, фолбэк последняя закупочная), списание (−qty, confirmed материальные расходы), остаток; фильтры: поиск, ед.изм, остаток (в наличии/на нуле), период поступления (серверный, влияет на поступление/сумму/списание); история приходов. **Цена закупки:** указывается при приходе (за единицу, необязательна); на материале хранится последняя закупочная (правка/удаление прихода пересинхронизирует её по самому свежему приходу с ценой); таблица показывает цену за ед. и сумму остатка (остаток × цена).
- **Расход по материалам (в сделке):** вкладка Финансы → «+ Расход» → «По материалам (со склада)»: выбор позиции с остатком, количество (живой пересчёт остатка); сумма считается АВТОМАТИЧЕСКИ = количество × закупочная цена (сервер пересчитывает, ручная сумма игнорируется); если у позиции цена не заведена (легаси) — сумма вручную. Сумма материального расхода при редактировании не правится. Чек не нужен, статус сразу confirmed; списывает остаток, при нехватке — ошибка; удаление расхода возвращает остаток. **Способ оплаты (нал/банк) выбирается при создании ЛЮБОГО расхода** (и материального, и прочего; у pending бухгалтер может поменять при подтверждении); плитки «прочие (нал/банк)» на Финансах материальные исключают явно. Блок счетов в карточке сделки называется «Аванс». На Складе колонка «Списание» кликабельна — модалка со списком сделок/заказов (ссылки), куда ушёл материал. Материал/количество после создания не редактируются (удалить и создать заново). Прочий расход менеджера/директора → статус **pending** («Ждёт бухгалтера»): финансистам задача «Подтвердить расход #N…» (срок 3 дня) + уведомление; бухгалтер/админ подтверждает в карточке сделки — выбирает оплату **нал/банк** и прикрепляет чек (без чека нельзя; чек менеджера при создании необязателен). После подтверждения задача закрывается, автор получает уведомление. В финансы/маржу/бонус входят только confirmed. Расход бухгалтера/админа подтверждается сразу.
- **Задачи:** доска; синхрон с сделок/цеха; срок дата+время, метка ПРОСРОЧЕНА; заметки; уведомление о просрочке (tasks:notify-overdue).
- **Финансы:** счета/платежи/расходы; маржа план/факт; менеджер видит и правит ТОЛЬКО свои (счета/платежи/расходы по чужим сделкам — 403). **Поступления денег** (cash_receipts, только financist/admin): сумма, нал (касса) / банк (счёт), откуда поступили, дата, комментарий, автор; таблица на Финансах с «+ Поступление» и удалением. **Сводка компании (эскиз бухгалтерии, минимализм):** верхний ряд — общая сумма договоров, дебиторка, остаток в кассе/банке = (платежи по счетам + ручные поступления) − расходы, всё по способу оплаты; Доход = по счетам + прочие поступления; ниже Доход (все поступления) − Расходы всего (разбивка: зарплата оклады+бонусы, налог, по сделкам/цеху, категории компании) = Чистая прибыль. **Расходы КОМПАНИИ без сделки** (кнопка «+ Расход компании», только financist/admin, категория обязательна, статус сразу confirmed, expenses.company_id): категории сидятся миграцией — Аренда, Комуслуги, Интернет, Бензин/ГСМ, Канцтовары, Хозтовары, Продукты питания, Офисные расходы, Налоги, Кредит, Прочее. Кредиторская задолженность НЕ отслеживается (нет источника данных). Раздел «Расходы»: сводка-плитки (все расходы с количеством, материальные (склад), прочие расходы (нал), прочие расходы (банк) — общая плитка «прочие» убрана, «ждёт бухгалтера»); плитки кликабельны и фильтруют таблицу; фильтр по периоду (с—по) влияет и на плитки, и на таблицу; таблица с фильтрами по виду, оплате, статусу и ссылками на сделку/заказ. Расход требует прикреплённый чек (фото/PDF, inline-просмотр); автор и дата заполнения фиксируются автоматически.
- **Зарплата:** К выплате = оклад + бонус − удержания + премии за выбранный месяц (селектор месяца в шапке). Оклад вводит financist/admin инлайн в ведомости (карандаш). **Корректировки** (payroll_adjustments, только financist/admin): отгул/больничный (можно днями — удержание авто = оклад/22 × дни), штраф и аванс (минус — аванс выдан вперёд и минусуется из ЗП), премия (плюс); видны в раскрытой строке сотрудника и менеджеру в своей карточке. Бонус по марже каждой сделки (ступени, см. §4); только успешные+оплаченные; менеджер видит своё + шкалу бонусов. **Роли-должности:** СЕО = admin, Финансист-Бухгалтер = financist; юрист/повар/дизайнер/технолог (lawyer/cook/designer/technologist) — права уровня «сотрудник» (цех, задачи, своя ЗП); после деплоя прогнать RolePermissionSeeder. Ступень выбирается по марже ДО налога — как «Маржа» на карточке сделки.
- **Дашборд слит с Аналитикой:** маршрут /dashboard остался (закладки, редирект после логина) — руководству редирект на /analytics, менеджеру — на сделки, остальным — в цех. Пункт меню убран. Блок «Последние сделки» удалён совсем.
- **Аналитика (admin/director/financist):** вкладка «Обзор» — фильтры (поиск №/контрагент/договор, период с—по, менеджер, этап; действуют на воронку, «за период» и топ менеджеров) → суммы в ДВЕ колонки-карточки (кол.1: общая сумма договоров = бюджеты ВСЕХ сделок кроме отменённых (не только won!), расходы, чистая прибыль; кол.2: оплачено, налог, ЗП-бонусы → Зарплата; каждая строка кликабельна) + блок «Деньги компании» (companyMoney, та же математика, что Финансы): остаток в кассе/банке, все расходы с разбивкой (ЗП, налог, по сделкам, категории), доход (счета + поступления), чистая прибыль полная + ряд долг клиентов / «за период» / конверсия → «требует внимания» (просроченные сделки → /deals/overdue, задачи → сделки, расходы «ждёт бухгалтера» → Финансы с exp_status=pending, материалы на нуле → Склад) → воронка АКТИВНЫХ сделок по этапам (клик → канбан) + топ-5 менеджеров за период (клик → сделки менеджера) → доход/расход по месяцам, статусы, ABC (по факт. оплате). Вкладка «По сотрудникам» — как раньше. Деньги — через PayrollService::companyTotals (единый источник с Финансами). AnalyticsController.
- **Чат** (общий/личный/групповой + чат сделки) — современный мессенджер: группы (название/описание/участники/фото), файлы и фото inline, поиск по чатам и внутри; **ответы-цитаты**, **редактирование своих** сообщений (метка «изменено»), автор под сообщением рядом со временем; **реакции-эмодзи**, **закреплённое сообщение** (баннер, закрепляет admin/director); **упоминания @имя** с уведомлением; **числовые счётчики непрочитанных** (серверный учёт, таблица chat_reads); **архив чатов** (по пользователю). Удаление ЛЮБОГО сообщения — только admin; своё — автор.
- **Сводный отчет** (/reports/deals, пункт меню после Чата, ТОЛЬКО admin/director — видны бонусы всех менеджеров): Excel-подобная таблица по всем сделкам компании (кроме отменённых): №, № договора/БИН, организация, адрес, товар, кол-во, сумма договора, оплачено, закуп (склад), прочие расходы, налог, остаток, маржа %, бонус менеджера, доход (экс-«фирма»: остаток − бонус), менеджер, срок, этап + плитки-итоги и строка Итого. Таблица ОБЩАЯ для сделок и цеха: у сделки с активным заказом цеха в колонке «Этап» второй бейдж «Цех · <этап цеха>» (цвет этапа цеха) — без дублирования денег. Формула денег — та же, что на карточке сделки/ЗП. Фильтры: поиск, период (по дате создания), менеджер, этап; строка кликабельна → карточка сделки. ReportController.
- **Аудит** (только admin — журнал по всем фирмам) / **Уведомления** (клик ведёт на связанную сделку/заказ) / **Доп.поля** / **Настройки**.

## 4. Бизнес-правила
- **Две компании (BAIA / ASU):** выбор при входе + переключатель в шапке; полное разделение данных (сделки, цех, финансы, аналитика, ЗП) — включая ПРЯМЫЕ ссылки (политики проверяют членство в компании: User::worksInCompany). Сотрудник может состоять в обеих; ЗП собирается со сделок каждой компании. Нумерация сделок по компании: BAIA-001 / ASU-001 (максимум числового суффикса строгого формата).
- **Воронки по компаниям:** у каждой фирмы СВОЯ воронка этапов (deal_stages.company_id; null = общий этап для тестов/легаси). Настройки → Этапы: выбор компании и вида (сделки | цех) прямо на странице. **Цех у КАЖДОЙ компании СВОЙ** (project_stages.company_id; BAIA — мебельный, ASU — швейный; null = общий для тестов): заказ попадает в цех компании своей сделки (ProjectStage::funnel), канбан цеха показывает воронку текущей компании, этап чужого цеха недоступен. DealStage::funnel/actStage/esfStage/wonStage/logisticsStage/workshopGateStage принимают companyId.
- **Режим «Все компании»** (кнопка «Все» в шапке, только admin/financist): session company_id=0 → общий отчёт по обеим фирмам (аналитика, финансы, ЗП); канбан показывает обе воронки с пометкой фирмы.
- **Поток этапов сделки:** Заключение договора → Дизайн и расчет → Закуп ЛДСП,МДФ (кнопка «В цех») → [цех: Кесу…Отправка, кнопка «Готово» возвращает на Логистику] → Логистика → Сборка → Акт утверждение → ЭСФ → Оплата успешно. **Спец-этапы ищутся ПО ТИПУ (deal_stages.stage_type):** contract/design/shop_gate/logistics/assembly/act/esf/payment_won (null = обычный); DealStage::actStage/esfStage/logisticsStage/wonStage/workshopGateStage читают тип — переименование этапа логику НЕ ломает; payment_won синхронизирует is_won. Тип уникален в пределах воронки. Настройки → Этапы: выбор воронки (BAIA/ASU/Цех), добавление/переименование/цвет/порядок (↑↓), удаление занятых этапов — с переносом сделок/заказов на выбранный этап; баннер предупреждает, если не назначены payment_won/shop_gate/logistics.
- **Роли по этапам:** менеджер ведёт сделку до «Акт утверждение» (включительно — отправка со Сборки на Акт); АКТ/ЭСФ/Оплата двигает ТОЛЬКО бухгалтер (financist) или admin (директор — нет); менеджер после АКТ видит сделку read-only (редактирование/смена ответственного/удаление тоже заблокированы). **Удаление сделки — ТОЛЬКО admin** (менеджер/директор/бухгалтер — нет, DealPolicy::delete); есть массовое удаление из вида «Список» (чекбоксы + «Удалить выбранные», DELETE deals/bulk — политика проверяется по каждой сделке). Завершение заказа цеха — ТОЛЬКО кнопкой «🚚 Готово → Логистика» (projects.toAct); перенос заказа на последний этап («Отправка») больше НЕ завершает его автоматически (с 15.07.2026) — заказ стоит на Отправке, пока цех не нажмёт кнопку.
- **Гейт-задачи (настраиваются на этапе, Настройки → Этапы):** у этапа задаются текст задачи, роль исполнителя (financist/manager/director/admin) и срок в днях; при входе на этап задача ставится КАЖДОМУ активному сотруднику роли + уведомление; пока не закрыта — сделка дальше не идёт; просрочка = уведомление (tasks:notify-overdue). Дефолт после миграции: «Акт утверждение» → «Выставить акт» (financist, 3 дня), «ЭСФ» → «Выставить ЭСФ» (financist, 30 дней). Галочка-гейт на карточке — только financist/admin.
- Успешная (won) = won-этап ИЛИ в цехе (есть проект) ИЛИ status=closed; отменённые — нет.
- **Доход** = только фактически поступившие платежи по успешным (won) сделкам. **Расход** = ВСЕ подтверждённые расходы компании считаются СРАЗУ (не ждут won) — потрачены деньги, видны во всех сводках немедленно (дашборд/финансы/аналитика). Бонус — от won-сделок.
- Счёт нельзя создать с суммой 0 (валидация min:1). Статус счёта (paid/partially_paid) — только пересчётом по платежам, вручную нельзя. Расход/счёт нельзя «переставить» на чужую сделку через update, статус самому себе не выставить.
- **Изоляция фирм по прямым ссылкам** во всех контроллерах денег (расходы/счета/платежи/склад) и поиске (binLookup, поиск цеха скоуплены по компании).
- **Бонус — ступенчатый от маржи КАЖДОЙ сделки** (маржа = остаток/бюджет, остаток = бюджет − налог − расходы): ≤10% → 0; 11–15% → 5%; 16–20% → 7%; 21–30% → 10%; 31–40% → 13%; от 41% → 15% от остатка (шкала от 15.07.2026). Считается по сделке, потом суммируется по менеджеру (PayrollService::bonusRateForMargin).
- Дедлайны: красный=просрочено, оранжевый=≤2ч. **Просроченной НЕ считается сделка на этапах ЭСФ/Оплата успешно** (по stage_type esf/is_won); на «Акт утверждение» просроченная сделка ПОКАЗЫВАЕТСЯ просроченной. Правило едино: /deals/overdue, счётчик «требует внимания», просроченные по сотрудникам, метка в Сводном отчёте. На /deals/overdue отдельной таблицей — **просроченные заказы цеха** (свой deadline у Project, скоуп по компании/ответственному).
- Финансы в цехе = финансы сделки.

## 5. Запуск
```
php artisan serve            # http://localhost:8000
npm run dev                  # или npm run build
php artisan schedule:work    # уведомления о просрочке
```
Первый вход — под учёткой администратора из сидера (`DatabaseSeeder`); **пароль сменить сразу после входа**. Деплой на сервер (Plesk + Git) и настройка `.env` — см. `DEPLOY.md`.
