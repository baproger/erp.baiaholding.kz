# BAIA ERP / CRM — Документация проекта

> Единый файл-справочник: суть проекта, что построено, как работает логика.

## 0. Суть проекта

**BAIA ERP** — внутренняя ERP/CRM для холдинга из двух фирм: **BAIA** (мебельное производство: металл + дерево) и **ASU** (швейный цех). Продакшн: `erp.baiaholding.kz` (Plesk). Заменяет Excel-файлы и тетради бухгалтерии.

Жизненный цикл денег и сделки:
1. **Предварительная сделка (лот)** — менеджер вносит цифры лота госзакупки, система считает маржу; ≥15% → «Подтвердить» → создаётся сделка.
2. **Сделка** идёт по воронке: Договор → Дизайн и расчет (гейт дизайнера) → Закуп ЛДСП,МДФ (уведомление снабженцу) → «В цех».
3. **Цех** (у BAIA два: Металл / Ағаш, у ASU один) — заказ идёт по этапам цеха с таймингом каждого этапа; «Готово» возвращает сделку на Логистику.
4. Логистика → Сборка → Акт → ЭСФ → **Оплата успешно** (двигает только бухгалтер).
5. **Финансы**: счета («Аванс»), платежи, расходы (по сделкам и компании), поступления, касса/банк, дебиторка/кредиторка. **ЗП**: оклад + ступенчатый бонус от маржи. **Аналитика/Сводный отчёт** — для руководства. **ТВ-экраны** в цехах и офисе — живые канбаны/рейтинги по коду доступа.

## 1. Технологии
- **Backend:** Laravel 13 (локально PHP 8.5, прод PHP 8.3 — composer.json `config.platform.php=8.3.30`), MySQL
- **Frontend:** Inertia.js + Vue 3 + Tailwind + Vite (Laravel Breeze)
- **Права:** spatie/laravel-permission (RBAC). **Admin — глобальный суперпользователь** (Gate::before, worksInCompany=true везде). Роль admin назначает только admin; последний активный админ защищён от разжалования/деактивации; админа редактирует/деактивирует только админ.
- **Таймзона:** Asia/Almaty (+05:00); **Локализация:** ru / kk
- **Тесты:** PHPUnit — 214+, все зелёные. Перед каждым push: `npm run build` + `php artisan test`.
- **Продакшн (Plesk):** сессии/кэш — файлы, очередь — sync; `vendor/` и `public/build/` закоммичены (на сервере нет composer/node); деплой — `DEPLOY.md` (Git Pull → `php artisan migrate --force` → `php artisan optimize`). CLI PHP на сервере без pdo_mysql — миграции гнать через Laravel Toolkit или веб-скрипт под FPM.

## 2. Роли и доступы
| Роль | Что видит / может |
|------|-----------|
| **admin** (СЕО) | всё; только он удаляет сделки и назначает роль admin |
| **director** | всё как наблюдатель; этапы АКТ/ЭСФ/Оплата не двигает |
| **financist** (Финансист-Бухгалтер) | финансы, аналитика, ЗП, создание сделок, сотрудники (просмотр/добавление/правка/деактивация, кроме админов), категории расходов, чек-лист предсделок, план продаж |
| **manager** | ТОЛЬКО свои сделки/лоты/цех/финансы/бонус; общей аналитики не видит |
| **employee (цех)** | карточки цеха (двигает этапы), свои задачи; суммы не видит |
| **lawyer / cook** | уровень «сотрудник»: цех, задачи, своя ЗП |
| **designer / supplier** («Технолог»→«Снабженец») | как сотрудник + просмотр сделок: подтверждают гейт-этапы («Дизайн и расчет» / «Закуп») |

## 3. Модули
- **Предв. сделки (/pre-deals):** лоты как Excel отдела продаж — № лота, БИН, заказчик, товар, контакт (имя+телефон), сумма/закуп/партнёр %/доставка/комиссия; авто-расчёт (PreDeal::calculate): партнёр=%×сумма, налог=ставка×сумма, остаток=сумма−всё, маржа=остаток/сумма (живой расчёт и в форме). Маржа ≥ порога (Setting `predeal_min_margin`, 15%) — «участвую: да» + «Подтвердить» → создаётся сделка (номер, бюджет, ответственный=менеджер, описание с расчётом), лот → confirmed со ссылкой; ниже — «нет/отклонена» (сервер блокирует). Менеджер видит только свои; руководство — все + фильтры + «Рейтинг менеджеров» (👑). Чек-лист лота (pre_deal_checklist_items: «КП через WhatsApp», «Позвонил клиенту»…) — галочки в раскрытой строке + телефон клиента; пункты настраивают admin/financist («⚙ Чек-лист»).
- **Сделки:** канбан+список (в списке длинные названия — line-clamp 2 строки); фильтр-бар (поиск, менеджер, этап — скрывает остальные колонки, срок/договор с—по); карточка с вкладками (Задачи/Финансы/Документы/Комментарии/Доп.поля/История/Чат). «Название сделки» убрано — везде company_name (name зеркалит его). Поля: компания BAIA/ASU, клиент, адрес, № договора (колонка bin, 100 симв.), дата договора, источник-портал, товар (client_name), кол-во (lot_number)+ед.изм., сумма, срок, заметка.
- **Цех (Projects):** канбан секциями по цехам; у BAIA два цеха (project_stages.workshop / projects.workshop: «Металл цех», «Ағаш цех»), у ASU один (workshop=null). «Отправить в цех» при 2+ цехах — модалка выбора; «Далее»/степпер/«Готово» — в воронке своего цеха. **Тайминг этапов** (project_stage_logs, хуки Project): бейдж «⏱ Nч Nм» на карточках (сколько заказ на текущем этапе) + блок «Тайминг этапов» на карточке заказа (история: каждый этап, когда вошёл/ушёл, сколько занял). Суммы от цеха скрыты.
- **ТВ-экраны (/screen, без логина, вход по 6-значному коду):** управление — Настройки → Экраны (коды по цехам + строка «Офис» на компанию, копирование кода, вкл/выкл, «Новый код» отключает старые — код сверяется на каждом показе; throttle 10/мин). **Экран цеха** — светлый канбан только своего цеха (без сумм, тайминг, часы, автообновление 30с). **Экран «Офис»** — рейтинг эффективности отдела продаж: балл = % от прибыли лучшего менеджера за месяц (прибыль по won-сделкам: бюджет−налог−расходы−бонус, как в ЗП; сами деньги СКРЫТЫ), средняя маржа %, успешные, конверсия, лидер 👑 справа крупно; фильтр месяца (кто был лучшим в любом месяце); «план сделок/мес» (Setting `sales_plan_monthly`, ставит admin/financist на стр. Экраны) — справочный индикатор, на лидерство не влияет.
- **Склад:** свой у каждой компании; приход (admin/financist) с ценой закупки; таблица материал/ед/цена/поступление/сумма/списание (кликабельно — куда ушло)/остаток; материальный расход в сделке = кол-во × закупочная цена (авто, confirmed сразу, списывает остаток).
- **Финансы:** «Аванс» (счета)+платежи, расходы сделок (pending → бухгалтер подтверждает с чеком и нал/банк) и расходы КОМПАНИИ по категориям (категории управляются «⚙ Категории»: добавить/переименовать/удалить, занятые — скрываются с сохранением сумм); Поступления денег (нал/банк, откуда) — сегодняшние + аккордеон прошлых с поиском/периодом; Расходы — так же (сегодня + прошлые аккордеоном); касса/банк = поступило − потрачено по способу оплаты; дебиторка (авто по счетам + вручную)/кредиторка (аккордеоны); удаление любой финзаписи → уведомление СЕО и директору; «Доход» = итог Сводного отчёта (по сделкам: остаток − бонус; при фильтре месяца — сделки месяца по дате договора); селектор «Месяц» пересчитывает Доход/Оборот/Расходы/Итог за месяц (ЗП и налог — только «за всё время»; касса/банк всегда текущие). Расходы > 60% суммы договора → уведомление финансистам (однократно).
- **Зарплата:** видны ВСЕ активные сотрудники (оклад/аванс можно дать любому). К выплате = оклад + бонус − удержания + премии за месяц; корректировки (отгул/больничный днями = оклад/22×дни, штраф, аванс, премия) вводит financist/admin; бонус ступенчатый от маржи каждой won-сделки × доля фактической оплаты (min(1, paid/budget)).
- **Аналитика (руководство):** LIVE-режим (часы, автообновление 60с), SVG-диаграмма доход/расход по месяцам со шкалой и линией итога (hover-детали), донат по статусам, спидометр конверсии, воронка, топ менеджеров, ABC, «требует внимания», «Деньги компании» (единая математика с Финансами через PayrollService/FinanceService). Вкладка «По сотрудникам».
- **Сводный отчет** (admin/director): Excel-таблица всех сделок+цеха (закуп/прочие/налог/остаток/маржа/бонус/доход), зелёный градиент won+Акт/ЭСФ, жёлтый Логистика/Сборка; период фильтрует по дате договора (fallback — дата создания) — как «Месяц» на Финансах.
- **Чат:** мессенджер (группы, файлы, цитаты, правка, реакции, закреп, @упоминания, счётчики непрочитанных, архив); чат сделки привязан к deal_id. Фоновые вкладки сервер не поллят. **Группы по фирмам:** у группы company_id (BAIA/ASU/обе) — сотрудник видит только группы своей фирмы (админ — все), проверка и по прямой ссылке. **Корзина:** удаление группы и личного чата — мягкое (сообщения целы), admin/director возвращают или стирают навсегда (с файлами); канал сделки удаляется только со сделкой. **Участники:** вкладка «Участники» — добавить сотрудника в готовую группу (для группы фирмы — только её сотрудники) / убрать (✕). **Звук** нового сообщения (WebAudio, тумблер 🔔 в localStorage) + живые бейджи через лёгкий поллинг /chat/state (4с на переднем плане; в фоне — только state раз в 30с + браузерное уведомление Notification API, работает и на телефоне при открытой вкладке). **Кто прочитал:** ✓/✓✓ на своих сообщениях (по chat_reads), в группах — счётчик и имена по наведению.
- **Сотрудники (/users):** карточки, сгруппированные по отделам; живой поиск, фильтр-чипы отделов, мини-статистика, скрытие отключённых; экспорт CSV (Excel). У сотрудника: день рождения (🎂 бейдж ≤7 дн., ежедневная команда users:notify-birthdays 09:00 уведомляет руководство и главу отдела), дата приёма (стаж «в компании с …»). **Профиль /users/{id}** (руководство или сам): контакты, договор, сделки, заказы цеха, задачи, ЗП-расчёт+корректировки (деньги — admin/financist/director и сам). **Отделы:** руководитель (⭐ на карточке; получает уведомления о просроченных задачах своего отдела через tasks:notify-overdue).
- **Просроченные:** сделки (без ЭСФ/won; Акт — считается) + отдельно заказы цеха. **Задачи, Документы, Аудит (admin), Уведомления, Доп.поля, Настройки (Общие/Этапы/Экраны/Доп.поля), Переводы.**
- Порядок меню: Аналитика → Просроченные → Сделки (перед ними Предв. сделки) → Цех → Склад → Сводный отчет → Финансы → Зарплата → Чат → … → Сотрудники → Профиль → Настройки.

## 4. Логика денег (единые формулы)
- **По сделке:** налог = бюджет × ставка (Setting tax_percent, 3%); остаток = бюджет − налог − confirmed-расходы; маржа % = (бюджет − расходы)/бюджет (ДО налога — по ней ступень бонуса); бонус = ставка(маржа) × остаток × min(1, оплачено/бюджет).
- **Ступени бонуса:** ≤10%→0; 11–15→5%; 16–20→7%; 21–30→10%; 31–40→13%; ≥41→15% (PayrollService::bonusRateForMargin).
- **Доход компании** = фактические платежи по won-сделкам; **расход** — все confirmed сразу; «Доход (Сводный отчёт)» = Σ(остаток − бонус) по не-отменённым сделкам. Прибыль для компании по сделке = остаток − бонус. Один источник правды: PayrollService (companyTotals/perUser/dealBreakdown) + FinanceService (companyBalances/dealsIncome) — Финансы, Аналитика, ЗП, Сводный отчёт и экран «Офис» сходятся.

## 5. Ключевые бизнес-правила
- **Две компании** BAIA/ASU: полная изоляция данных (worksInCompany в политиках, прямые ссылки тоже), свои воронки сделок и цехов (Настройки → Этапы: выбор компании и вида; у цеха — подвкладки по цехам, порядок 1..N внутри каждого). Режим «Все» (admin/financist) — сводно по обеим.
- **Спец-этапы по stage_type** (contract/design/shop_gate/logistics/assembly/act/esf/payment_won) — переименование не ломает логику; тип уникален в воронке.
- **Гейт-задачи этапов:** текст/роль/срок на этапе; при входе — задача каждому активному сотруднику роли + уведомление; сделка дальше не идёт, пока не закрыта. Галочку ставит роль гейта или admin. «Дизайн и расчет» → дизайнер (блокирует «Закуп»); «Закуп» → снабженец (отправку в цех не блокирует); АКТ/ЭСФ → бухгалтер.
- **Роли по этапам:** менеджер до Акта; АКТ/ЭСФ/Оплата — только financist/admin; после Акта менеджеру read-only. Оплата успешно НЕ требует полной оплаты (остаток — в дебиторку). Удаление сделок — только admin (+bulk); каскад: заказ цеха отменяется, ссылки помечаются «сделка удалена».
- **Нумерация сделок:** BAIA-001/ASU-001 по компании; считаются только ЖИВЫЕ (при удалении номер освобождается: `…#del{id}`); удалили все — снова с 001; миграция 2026_07_21 разово перенумеровала живые с 001 (чаты по deal_id — история цела).
- Счёт с суммой 0 запрещён; статусы счетов — только пересчётом платежей; расход/счёт не переставить на чужую сделку.
- Просрочка: ЭСФ/won исключены, Акт — просрочен; и сделки, и заказы цеха.

## 6. Скорость и безопасность
- **Перф:** OPcache на проде (Plesk → PHP Settings) — обязателен; после каждого деплоя `php artisan optimize`; индексы под горячие запросы (миграция add_performance_indexes); чат/экраны не поллят из фоновых вкладок; статика с `Cache-Control: immutable, 1y` (public/.htaccess, имена версионированы); brotli на сервере; код-сплит по страницам.
- **Безопасность:** SecureHeaders (nosniff, X-Frame-Options, Referrer-Policy, Permissions-Policy, HSTS), принудительный HTTPS в production, троттлинг логина и публичных форм, скрытые файлы в public/ не отдаются (403), прод `.env`: APP_DEBUG=false, SESSION_SECURE_COOKIE=true. Аудит-лог, уведомления об удалении финзаписей. Рекомендовано владельцу: регулярные бэкапы БД в Plesk, ротация FTP-пароля после любых передач.
